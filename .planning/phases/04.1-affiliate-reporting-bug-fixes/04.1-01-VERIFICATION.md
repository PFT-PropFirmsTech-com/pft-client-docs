### Plan Deviation Acknowledged

SUMMARY documents that Bugs 2 + 3 were closed by remote hotfix commits (`7110ce54`, `bc0ef4ec`, `31ab6d1b`) with a different architectural shape than the plan specified:

- Plan proposed page-level `formatCurrency(amount, currency?)` overload + row's `r.currency` passthrough.
- Remote shipped a separate `formatPurchaseRowCurrency` helper + USD-canonical render + `r.payment?.usdAmount ?? derivedUsd` for purchase amount.

Both close the audit symptom. Executor honored `<deviation_rules>` Rule 4 (defer to deployed-style approach rather than overwrite). For verification purposes both shapes satisfy must_have truths #3 and #4 — outcome-equivalent.

Plan proposed regex-based `parseAccountSize` with k=1e3/M=1e6 multipliers; remote shipped strip-commas + `Number.isFinite` + raw-string fallback `formatAccountSizeLabel`. Both prevent `$NaN`. The remote shape will NOT expand `"25k"` → `$25,000` though — for an `accountSize` value of `"25k"`, `formatAccountSizeLabel` returns `$25k` (since `Number("25k")` is NaN, falls through to `$${str}`). The plan's regex would have returned `$25,000`. This is a **minor semantic divergence** worth flagging in the human-verify pass: if Trading Cult `accountSize` values are stored as raw numeric strings (`"5000"`, `"200000"`), the remote shape produces `$5,000` / `$200,000` correctly. If they're stored with `k`/`K` suffixes, the row will render `$25k` instead of `$25,000` — still readable, never `$NaN`, so the audit's bug bar is met either way.
