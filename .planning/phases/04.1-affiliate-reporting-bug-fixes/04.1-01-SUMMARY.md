**Bug 3 (`$NaN`) — SOLVED differently on remote.** Remote adds module-scope `formatAccountSizeLabel(size)`:

```typescript
const formatAccountSizeLabel = (size?: string | number | null): string => {
  if (size === null || size === undefined || size === "") return "";
  const str = String(size).trim();
  if (str.startsWith("$")) return str;
  const n = Number(str.replace(/[\s,_]/g, ""));
  return Number.isFinite(n) ? `$${n.toLocaleString()}` : `$${str}`;
};
```

Different from the plan's proposed `parseAccountSize` (regex `/^([\d.]+)\s*([kKmM]?)/` with k=1e3 / M=1e6 multiplier). Remote handles "$200,000" / "25k" / "200k" via strip-commas-then-Number + raw-string fallback. Same outcome: `$NaN` cannot render. Both `Number(r.payment.accountSize)` sites (formerly L178, L743) replaced with `formatAccountSizeLabel` calls.

**Bug 2 (Commission Amount currency) — SOLVED differently on remote.** Remote forces USD-canonical rendering rather than the plan's row-currency approach:

```typescript
// PurchaseReportTable cell render (line 259 post-merge):
{formatPurchaseRowCurrency(r.amount, "USD")}
```

Plan proposed passing `r.currency || r.payment?.currency` (row currency); remote chose to force `"USD"` everywhere because commission amounts are USD-canonical in storage. Same outcome on the live audit symptom (¥1 / ¥3 on Trading Cult → real USD figures like `$0.69`). Remote also: derives `purchaseAmount` as `r.payment?.usdAmount ?? (r.amount / (r.rate/100))` and forces `purchaseCurrency = "USD"` so purchase × rate% == commission is internally consistent (audit had also noted "Purchase Amount in row currency" — remote chose a stricter "everything USD" stance that arguably handles mixed-currency cases more robustly than the plan).

**Reversed-commission UI bonus.** Remote `31ab6d1b` also adds a "Reversed" tag + `line-through` styling on reversed entries — was not in the audit/plan but is a related correctness improvement.

**Resolution.** After the initial commit `80e4157f` (which carried the plan's Bug 1 + Bug 2 + Bug 3 shape) hit a rebase conflict on `AffiliatesContainer.tsx`, the rebase was aborted, the commit was soft-reset, the `AffiliatesContainer.tsx` changes were reverted via `git checkout --`, the remote was fast-forwarded, and only the Bug 1 patch to `usePayments.ts` was re-committed (`60e9b37c`). **After** (lines 338-376, post-commit `60e9b37c`):
```typescript
const commissions: any[] = commByOrder[p._id] || [];
const directTier =
  commissions.length > 0
    ? [...commissions].sort((a, b) => (a.tier ?? 1) - (b.tier ?? 1))[0]
    : null;
const totalCommissionAmount = commissions.reduce(
  (sum, c) => sum + (Number(c?.amount) || 0),
  0,
);
// ...
escapeField(directTier?.rate ?? ""),
escapeField(commissions.length > 0 ? totalCommissionAmount : ""),   // <- SUM across all tiers
escapeField(directTier?.currency ?? ""),
escapeField(directTier?.affiliateUserId ?? ""),
```

Grep evidence (post-push):
- `grep -n "Direct Commission Rate" src/hooks/usePayments.ts` → `325: "Direct Commission Rate (%)",`
- `grep -n "totalCommissionAmount" src/hooks/usePayments.ts` → `348` (def) + `373` (use)

Behavior:
- Multi-tier MLM payment: amount column = sum across all `commByOrder[p._id]` entries (e.g. tier 1 $1.62 + tier 2 $0.81 + tier 3 $0.40 = $2.83), rate column = tier-1 direct rate, affiliate identity columns = tier-1 direct referrer.
- Single-tier (non-MLM) payment: reduce of length-1 array = same single value as before — backward compatible.
- No commissions for a payment: amount column blank (not "0") — avoids confusing zero rows.
