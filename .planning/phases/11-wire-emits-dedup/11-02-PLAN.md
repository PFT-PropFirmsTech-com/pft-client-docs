<objective>
Wire the purchase/conversion tracking events at every real payment-completion path, threading the partner clickid, normalized USD payout, and an FTD flag — and guarantee exactly ONE partner-conversion-eligible event per PAP purchase.

Purpose: CRM-05 + CRM-06. Phase 12's partnerPostback needs a `purchase_completed` (standard) or `pap_payment_completed` (PAP) event carrying `partnerClickId` + `usdAmount` + `currency:USD` + `isFirstPurchase`, resolvable even in a `req=null` gateway callback. Today `TrackingEvents.purchaseCompleted` has ZERO callers and the existing `papPaymentCompleted` calls pass the BILLED amount (JPY-as-USD bug risk) with no clickid.

Output: a new `emitTrackingPurchaseCompleted(payment)` utility, called at the standard completion sites (free-$0 + paid callback + paid stripe + fanbasis); the two existing `papPaymentCompleted` calls fixed to use `usdAmount` + partnerClickId + a stable eventId; a hard guard that PAP funded-legs emit ONLY `pap_payment_completed`, never `purchase_completed`.
</objective>

<critical_design>
Read before coding — this prevents a checker false-flag and a double-count bug.

1. **purchase_completed is NOT suppressed on repeat purchases.** It fires on EVERY standard purchase (Meta CAPI / GA4 / Klaviyo need all of them). FTD = the `isFirstPurchase` boolean flag on the payload, nothing more. The "conversion fires once per user" rule is enforced in Phase 12 (partnerPostback gates on `isFirstPurchase===true`). Do NOT add any "skip 2nd purchase_completed" logic.

2. **Emit-site reality (verified):** `SalesWebhookDispatcher.purchaseCompleted` / `emitPurchaseCompletedWebhook` fires at exactly ONE site today — `payment.service.modular.ts:1507` (the free-$0 path, which also covers free/100%-coupon PAP funded-legs). PAID standard completions in callback.service.ts / stripe-webhook.service.ts / fanbasis-webhook.service.ts do NOT emit purchase_completed today. So this plan ADDS the tracking purchase emit at each paid completion too. `payment.model.ts:338` post-save does ONLY Prometheus metrics (recordPayment) — it does NOT emit purchase, so there is NO purchase_completed double-emit via post-save.

3. **Exactly one conversion-eligible event per PAP purchase.** PAP funded-legs already emit `pap_payment_completed` (callback:854, stripe:507). To avoid a PAP purchase ALSO emitting `purchase_completed` (partner would double-count), `emitTrackingPurchaseCompleted` MUST early-return when the payment is a PAP funded-leg (`payment.payAfterPass === true && payment.currentProgramId`). PAP's single conversion-eligible event is `pap_payment_completed`.

4. **usdAmount, never billed currency.** Payout value = `payment.usdAmount` with `currency:"USD"` ALWAYS. The existing papPaymentCompleted calls pass `value: payment.payAfterPassRemainingPrice` — that is the BILLED amount (can be JPY). Switch to `usdAmount`.

5. **partnerClickId source = `payment.attribution?.partnerClickId`** (Phase 10 persisted it there for the req=null case). NEVER a new User.findById at purchase sites.

6. **Stable dedup key.** `deterministicEventId` buckets by MINUTE, so a gateway retry a minute later would NOT dedup. Pass an explicit stable `eventId`: `purchase:<paymentId>` for purchase_completed, `pap:<paymentId>` for pap_payment_completed. The dispatcher honors a provided `payload.eventId`.

7. **FTD timing (verified):** the payment is flipped to `status:"completed"` (via `payment.save()` at callback:716 / `updateOne status:completed` on the free path / `payment.status="completed"; save()` in stripe+fanbasis) BEFORE the completion side-effects (emails, XP, emit) run. So at emit time, `Payment.countDocuments({ userId, status:"completed" }) === 1` means THIS is the first purchase → `isFirstPurchase:true`. Use `=== 1`, not `=== 0`. If any specific call site emits BEFORE the status flip, count `=== 0` there instead — but pin it per-site by reading the surrounding code; do not assume.
</critical_design>
