From the code, document the dedup contract:
  - `TrackingEventLog` has a compound unique index on `(eventId, destination)`; the dispatcher (`tracking.service.ts`) pre-checks `isDuplicate(eventId, destination)` and `reserveLogRow` (insert-on-conflict → 11000 → treated as deduped) before calling each adapter. So the SAME eventId dispatched twice to the same destination resolves to ONE send.
  - CRITICAL nuance to record: when a caller does NOT supply `payload.eventId`, the dispatcher falls back to `deterministicEventId(user, eventName, timestampMs)` which buckets by MINUTE (`Math.floor(ts/60000)`). That default dedups only WITHIN the same minute — a gateway webhook retry a minute+ later would generate a DIFFERENT eventId and fire again. Therefore durable cross-minute dedup REQUIRES an explicit stable eventId.
  - Confirm from the 11-01 / 11-02 SUMMARYs (and by grepping the wired call sites) that Phase 11 passes STABLE eventIds:
    - `signup:<userId>` at both registration sites (11-01) → repeat OTP submit dedups.
    - `purchase:<paymentId>` at standard completion sites (11-02) → duplicate provider webhook / retry dedups.
    - `pap:<paymentId>` at PAP completion sites (11-02) → PAP retry dedups.

  - Note the residual: dedup is PER-DESTINATION. Once Phase 12 adds the `partnerPostback` destination, the SAME stable eventId dedups the partner postback too — so a retried PAP webhook cannot double-send the conversion postback. Record this as the forward-looking guarantee that closes CRM-08 end-to-end (the FTD once-per-user gate is separate, enforced in Phase 12 via isFirstPurchase).
