<objective>
CRM-03: Persist the partner `clickid` on the Payment document at checkout creation, so a later purchase completed via a gateway/webhook callback — where `req = null` and no cookie exists — can still resolve the clickid off the Payment doc. Add `partnerClickId` to the `PaymentAttribution` interface + the `attribution` subdocument in the Payment schema, and populate it at BOTH payment-creation sites from the AUTHORITATIVE source: the authenticated user's User doc (`user.partnerClickId`, set at registration by plan 10-02).

Purpose: `enrichClickIds` (Tracking/enrichment/click-ids.ts) reads only `req.cookies`/`req.query` and covers ad-platform IDs only; gateway completion paths run with `req = null`. The User doc is the durable, authoritative carrier. At checkout the user is authenticated and the service already loads the User doc before creating the Payment — so read `user.partnerClickId` there. This plan covers BOTH the standard-challenge checkout AND the PAP funded-leg path (PAP counts as a conversion per the milestone), so PAP first-purchases carry the clickid too.

Output: `attribution.partnerClickId` stored on standard and PAP Payment docs when the buyer's User doc has one; absent otherwise.
</objective>
