## Milestone Summary

**Key Decisions:**

- Bulk-by-orders endpoint uses **POST body**, not GET query string — 2000+ orderIds would exceed nginx/browser URL length limits at ~8KB. Set as the reusable pattern for future bulk exports.
- `/affiliates/my-commissions` uses `Auth(userRole.user)` ONLY — never include admin/backOffice/sales. Service scopes to `req.user._id` derived from the auth context, never accepts an override userId from query/body. Prevents any future IDOR regression.
- Lowest-tier (tier 1 = direct referrer) commission picked per payment when MLM has multiple tier entries — keeps a deterministic single row per payment in the CSV.
- `skipEnrichment=true` preserved on the base payment fetch — commission data merged via second bulk call, not via per-row enrichment. Keeps existing export performance unchanged on hundreds of rows.
- Commission fetch failure inside `useExportPaymentsCsv` degrades to empty affiliate columns (not a broken export). Existing 19 columns always ship.
- Purchase Report card scoped to `{showOverview && ...}` block only — sibling `{showWithdrawals && ...}` deliberately untouched. Single non-conditional `useGetMyCommissions` hook call (Rules of Hooks safe).
- `PurchaseReportTable` declared at module scope, NOT inside `AffiliatesContainer`. Avoids accidental component remounting on every parent render.
- `payment.mt5Login` confirmed present on all linked TradingCult payments (16/16) — no need to traverse user.programs[].mt5AccountId; backend reads it directly from Payment.
- Commission `currency` field rarely set on `affiliatecommissions` rows (1/22 in live TradingCult DB). Fall back to `payment.currency` for accurate display/CSV value.
- Ticket cmqqchwh kept at IN_PROGRESS after the clarification reply (NOT WAITING_CLIENT) because items 1 + 3 from the same ticket are still in flight — flipping mid-ticket would force partial confirmations.

**Issues Resolved:**

- Affiliate commission attribution was invisible in admin payment exports (couldn't reconcile customer purchases against affiliate payouts). Now exposed in the existing CSV with 6 added columns.
- Affiliates had no per-purchase commission visibility — only cumulative totals on "Your Referrals" rows. Now have a row-per-commission Purchase Report filterable by tier + exportable.
- Client confusion over Payout vs Withdrawal History terminology — clarified on the ticket.

**Issues Deferred:**

- Live human-verify of the Purchase Report card visual + Export CSV download + Network panel single-POST confirmation — deferred until next main-2026 deploy (6-item checklist in `04-VERIFICATION.md`). Same pattern as v1.0 Phases 2/3.
- v1.0 pending human-verify checklists (Phases 2 & 3) also still deferred from v1.0 — same deploy unlocks both.
