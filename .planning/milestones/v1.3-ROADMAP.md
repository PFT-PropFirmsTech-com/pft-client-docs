# Milestone v1.3: CRM Partner Tracking (S2S Postbacks)

**Status:** ✅ SHIPPED 2026-07-01 (code-complete + code-level verified; live human-verify deferred pending deploy + partner config)
**Phases:** 10-12
**Total Plans:** 9 (+2 deferred human-verify checkpoints)

## Overview

A Trading Cult affiliate partner can attribute registrations and conversions to their own traffic — a partner `clickid` is captured at landing, persisted through signup and purchase, and fires S2S GET postbacks (with clickid + goal + payout) on registration and first sale (FTD only). Backend-heavy (pft-backend) + small pft-dashboard cookie-capture piece. Source ticket: cmqt52jdb001dny0kknkou9x0.

**Hidden prerequisites made explicit at milestone start:**
- `TrackingEvents.signupCompleted()` and `.purchaseCompleted()` were defined but had ZERO callers — Phase 11 wired both as new call sites.
- The existing `conversionWebhook` adapter was NOT reusable (wrong event map + POST/JSON/HMAC vs required GET/macro) — Phase 12 built a new `partnerPostback` adapter.

## Phases

### Phase 10: Capture & Persist

**Goal:** The partner `clickid` is captured at the tracking-link entry point, survives in a first-party cookie to the registration form, and is persisted durably on both the User document and the Payment document — so every downstream event emitter can resolve it even from a gateway webhook callback where no browser request exists.

**Depends on:** Nothing (first v1.3 phase; builds on existing User/Payment schema patterns)
**Requirements:** CRM-01, CRM-02, CRM-03
**Plans:** 4 plans (3 autonomous code plans in Wave 1 + 1 deferred live-verify checkpoint in Wave 2)

Plans:
- [x] 10-01: `GET /track` route + `_partner_clickid` cookie (backend `09ca7387`) + pft-dashboard cookie-read + forward as `partnerClickId` in signup body (`e111dab1`) (CRM-01)
- [x] 10-02: `partnerClickId` on User interface + indexed schema field; persists via existing registration payload spread, survives OTP round-trip — no auth.service.ts edit (backend `d2992553`) (CRM-02)
- [x] 10-03: `partnerClickId` on Payment `attribution` interface+schema; persist from authoritative `user.partnerClickId` at standard checkout AND PAP funded-leg creation (backend `4a079169`) (CRM-03)
- [ ] 10-04: DEFERRED post-deploy live human-verify of the full capture→persist path (CRM-01/02/03)

**Success Criteria** (all met, code-verified):
1. `GET /track?clickid=ABC123` sets a `_partner_clickid` first-party cookie and redirects; value unchanged.
2. Post-tracking-link registration → `partnerClickId: "ABC123"` on the User document.
3. That user's payment → `partnerClickId: "ABC123"` on the Payment document (checkout creation, before any gateway callback).
4. Organic registration (no tracking link) → no `partnerClickId` field (skip-when-absent).

**Details:**
Partner-facing tracking URL is `/api/tracking/track?clickid=…` (mounted under the existing tracking router). A prettier `/track` requires a brand-landing infra rewrite — config/verify detail, not code.

### Phase 11: Wire Emits + Dedup

**Goal:** The two tracking helpers that had zero callers (`TrackingEvents.signupCompleted()` and `.purchaseCompleted()`) are wired at their real call sites — registration completion and all payment-completion paths including the PAP path — threading `partnerClickId` + `usdAmount` + `currency=USD` through each; an FTD (first-purchase) FLAG is produced on every purchase event; and a dual-dispatch audit ensures the legacy `ConversionWebhookEventsService` path cannot double-fire the same event.

**Depends on:** Phase 10 (partnerClickId must be on User + Payment docs before emit sites can read it)
**Requirements:** CRM-04, CRM-05, CRM-06, CRM-08
**Plans:** 3 plans (11-01 Wave 1 = shared type extension + signup wiring; 11-02 + 11-03 Wave 2, parallel — disjoint files)

Plans:
- [x] 11-01: Extend `ITrackingEventPayload` + helper signatures with `partnerClickId`/`isFirstPurchase`/stable `eventId`; wire `TrackingEvents.signupCompleted()` at both registration-completion sites (backend `8e2f7509`+`44deb3d4`) (CRM-04)
- [x] 11-02: `emitTrackingPurchaseCompleted` utility (attribution.partnerClickId + usdAmount + FTD flag + PAP-skip + stable eventId) wired at 4 standard completion sites; `papPaymentCompleted` at 3 sites switched to usdAmount (JPY bug) + partnerClickId + stable eventId; fanbasis DOES provision PAP → wired there too (backend `644ccd39`+`982ba9a1`) (CRM-05, CRM-06)
- [x] 11-03: CRM-08 dual-dispatch audit — legacy `ConversionWebhookEventsService` (6 methods, KYC/payout/challenge only) confirmed disjoint from signup/purchase/pap; stable eventIds make retries idempotent → `11-DEDUP-AUDIT.md` (`8540f5a`) (CRM-08)

**Success Criteria** (all met, code-verified):
1. Registration (one-step OR OTP) → `signup_completed` fires with `partnerClickId`; repeat submission dedups (stable `signup:<userId>` eventId).
2. Completed payment (standard or PAP) → purchase event fires with `partnerClickId`, `usdAmount`, `currency:"USD"` — resolved from `payment.attribution.partnerClickId` (works with `req=null`).
3. Every purchase event carries `isFirstPurchase` boolean. `purchase_completed`/`pap_payment_completed` NOT suppressed on repeat purchases (Meta/GA4/Klaviyo need every one) — once-per-user conversion gate lives in Phase 12's adapter.
4. PAP funded-leg fires exactly ONE partner-conversion-eligible event (`pap_payment_completed`, `usdAmount` not billed `payAfterPassRemainingPrice`) — never also `purchase_completed` for the same payment. Free $0 purchase → no conversion-eligible event.
5. Retries idempotent via stable `purchase:<paymentId>`/`pap:<paymentId>` eventIds (default `deterministicEventId` only dedups within a minute).
6. Dual-dispatch audit: `ConversionWebhookEventsService` fires only KYC/payout/challenge — disjoint from new Tracking signup/purchase events, no double-fire.

## Milestone Summary

**Key Decisions:**
- `partnerClickId` sourced from `payment.attribution.partnerClickId` at purchase sites (not a new `User.findById` lookup) — Phase 10 put it there specifically so `req=null` gateway callbacks can resolve it.
- FTD = `isFirstPurchase` boolean flag on every purchase event, NOT event suppression — `purchase_completed`/`pap_payment_completed` still fire for every purchase (Meta/GA4/Klaviyo need them); the once-per-user conversion gate lives in the Phase 12 adapter.
- Payout = `usdAmount` + `currency=USD` always — guards the known JPY-billed-as-USD bug class; the existing `papPaymentCompleted` was passing billed `payAfterPassRemainingPrice` and got fixed at all 3 sites.
- Stable eventIds (`signup:<userId>`, `purchase:<paymentId>`, `pap:<paymentId>`) — the default `deterministicEventId` is minute-bucketed, insufficient for gateway-webhook retries arriving minutes later.
- New `partnerPostback` adapter instead of reusing `conversionWebhook` — wrong wire protocol (POST/JSON/HMAC vs required GET/macro) and wrong event map.
- Minimal one-off config for Trading Cult (`TrackingSettings.destinations.partnerPostback`), not a generic multi-partner admin UI — build reusable only when a 2nd partner appears.
- Config disabled + empty URLs by default — no brand fires unless explicitly configured; zod validation block added so `PUT /api/tracking/settings` doesn't silently strip the new field.
- Defer-to-remote convention applied twice mid-milestone (unrelated remote commits fast-forwarded cleanly, zero file overlap).

**Issues Resolved:**
- Two "wired" tracking helpers (`signupCompleted`/`purchaseCompleted`) that actually had zero callers — now fire at every real completion path.
- JPY-as-USD payout bug pre-empted at all 3 PAP payment-completion sites before ever reaching a partner payout field.
- `fanbasis-webhook.service.ts` PAP provisioning path discovered mid-Phase-11 (not covered by the milestone's original research) and wired correctly.

**Issues Deferred:**
- Live human-verify of the full capture→persist→emit→postback chain against a real Trading Cult config + real partner endpoint (10-04 + Phase 12 SC5) — both gated on the next main-2026 deploy AND the partner's URL templates.
- No dedicated cross-phase milestone audit run — each phase's plan-checker + goal-verifier already cross-checked its inputs against the prior phase's outputs (Phase 11 verified it reads Phase 10's fields; Phase 12 verified it reads Phase 11's fields).

**Technical Debt Incurred:**
- None identified — 12-03's behavioral harness found zero adapter bugs; whole-chain scoped tsc clean.
