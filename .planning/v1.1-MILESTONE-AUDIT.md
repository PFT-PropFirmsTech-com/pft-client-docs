## 2. Reproduction (live code, no speculation)

### Bug 1 — CSV tier-1-only

[`pft-dashboard/src/hooks/usePayments.ts:325–360`](pft-dashboard/src/hooks/usePayments.ts#L325)

```typescript
// For MLM payments with multiple tier entries, take the lowest tier (tier 1 = direct).
const c = commissions.length > 0
  ? commissions.sort((a, b) => (a.tier ?? 1) - (b.tier ?? 1))[0]
  : null;
```

**Recorded as intentional in `.planning/milestones/v1.1-ROADMAP.md` Key Decisions:**

> "Lowest-tier (tier 1) commission picked per payment for CSV row — Deterministic single-row-per-payment in CSV; full breakdown still available via bulk endpoint"

That was MY scope call at plan time. Client explicitly disagrees: they want **TOTAL** across all tiers — not "deterministic single row". The single-row-per-payment goal is right; the value should be SUM(tiers), not tier-1-only.

### Bug 2 — Commission Amount renders wrong figure + wrong currency

[`pft-dashboard/src/app/(dashboard)/_components/modules/users/affiliates/AffiliatesContainer.tsx:407–415`](pft-dashboard/src/app/(dashboard)/_components/modules/users/affiliates/AffiliatesContainer.tsx#L407) — page-level helper:

```typescript
const currencyCode = activeSettings?.currency || "USD";  // brand currency: JPY on TradingCult

const formatCurrency = (amount: number) => {
  const formatted = new Intl.NumberFormat("en-US", {
    style: "currency",
    currency: currencyCode,           // ← HARDCODED brand currency
    currencyDisplay: "symbol",
  }).format(amount);
  return formatted;
};
```

[`AffiliatesContainer.tsx:743`](pft-dashboard/src/app/(dashboard)/_components/modules/users/affiliates/AffiliatesContainer.tsx#L743) — Purchase Report row (shipped by Phase 4):

```typescript
{r.amount != null ? formatCurrency(r.amount, r.currency) : "—"}
```

The plan author (me) called `formatCurrency(r.amount, r.currency)` assuming a 2-arg signature. The actual function takes ONLY `amount` and silently discards the second arg. So `r.currency` (the commission's own currency, often `USD`) is ignored and Trading Cult's brand currency (JPY) is forced.

Commission `amount` in MongoDB is stored as the USD-equivalent decimal value:

```js
// live TradingCult.affiliatecommissions sample (from earlier session DB query)
{ "amount": 1.25, "rate": 10, "tier": 1, ... }  // USD value, no currency field
```

`Intl.NumberFormat("en-US", { style: "currency", currency: "JPY" }).format(1.25)` → **`"¥1"`** (JPY uses 0 decimal places, rounds to integer). Reproduces the screenshot exactly.

### Bug 3 — Product `twoStep $NaN`

[`AffiliatesContainer.tsx:178`](pft-dashboard/src/app/(dashboard)/_components/modules/users/affiliates/AffiliatesContainer.tsx#L178) (CSV export builder) and [`:743`](pft-dashboard/src/app/(dashboard)/_components/modules/users/affiliates/AffiliatesContainer.tsx#L743) (table render):

```typescript
const size = r.payment?.accountSize
  ? `$${Number(r.payment.accountSize).toLocaleString()}`
  : "";
```

`accountSize` is stored as a string in the Program collection: `"5k"`, `"10K"`, `"100k"`, `"500k"`, `"5000"`. `Number("5k")` → `NaN`. Renders as `twoStep $NaN`.

Fix shape: `parseAccountSize(s)` regex `/^([\d.]+)\s*([kKmM]?)/` → multiplier `k=1e3 / M=1e6`. The repo already does this in [`pft-backend/src/app/modules/Admin/Program/accountSize.utils.ts`](pft-backend/src/app/modules/Admin/Program/accountSize.utils.ts) (`parseAccountSizeValue`) — added in v1.1 for the flexible-PAP work. Mirror on the dashboard side.

## 3. Why the v1.1 verifier missed all three

The Phase 4 plan-checker passed with the note:

> "Commission Amount cell renders the wrong figure" was logged as Truth #4 in [`04-VERIFICATION.md`](.planning/phases/04-affiliate-reporting/04-VERIFICATION.md) only in the deferred `human_needed` checklist (post-deploy visual check), not as a static must-have. **Static checks confirmed the diff matches the plan; the plan itself was wrong.**

Tier-1-only was an explicit Key Decision — the verifier rightly didn't flag a passing design decision. The client disagreement only surfaced when they SAW the live behaviour and rejected the assumption.

The `$NaN` rendering was invisible to grep/tsc because `Number("5k")` is valid TypeScript that produces NaN at runtime. No static check would have caught it; only an actual render against a real DB sample would.

This validates the GSD pattern: **`human_needed` checkpoints exist because not every failure can be caught statically**. The reopen is the system working — we caught the gaps before the client did business on bad data.

Option (b) keeps the page-level helper untouched (lower blast radius).
